# 設計検証

## 前提

実装前の設計検証は、実装後の手戻りを防ぐ最もコスト効率の高い投資である。
AIが書くコードに「メンタルモデル」は存在しない。実装者（人間）が設計を理解し、判断できる状態を作ってから着手する。

「コードを書く速度」ではなく「正しい方向に書き始める確度」を最大化する。

## 検証の重点領域

以下のいずれかに該当する領域を重点的に検証する。

- **スコープ**: 複数ファイルにまたがる実装、新規エンティティ/テーブル/APIエンドポイントの追加
- **ドメインリスク**: 状態遷移を伴う機能、外部システム連携、認証・認可・決済の変更、データの不可逆な変更
- **新規性**: プロジェクトで初めて導入するパターン、既存アーキテクチャにない層の追加
- **ビジネスロジック密度**: 条件分岐が多い業務ルール、複数ステークホルダーの要件が絡む機能

## 実行手順

### 1. 仕様の取得と整理

- $ARGUMENTS に issue 番号や URL があれば内容を取得する
- 仕様書・要件・ユーザーフローなど、与えられた情報をすべて読み込む
- 情報が不足していると感じたら、この時点で AskUserQuestion で確認する

### 2. 対話による深掘り

AskUserQuestion を使って設計上の論点を1つずつ深掘りする。
まず大きな認識ズレがないか確認する質問から始め、段階的に詳細な論点に進む。
質問の回数に制限はない。ユーザーの回答から新たな論点が見えたら、継続的にインタビューする。

以下の観点で質問を組み立てる。

#### A. ドメイン境界

- 既存のエンティティに別ドメインの情報を載せようとしていないか
- 「この情報はどのエンティティが持つべきか」に迷う箇所はないか
- システム間連携がある場合、境界はどこか

#### B. 状態遷移

- 逆方向の遷移（ステータス戻し）は想定されているか
- 同じ遷移が2回以上発生した場合の挙動は定義されているか（冪等性）
- 遷移に伴う副作用（API呼び出し、メール送信）は1回だけ実行されるか

#### C. 要件の曖昧さ

- 「〇〇の場合」とあるが、「〇〇でない場合」はどうなるか
- 数量・頻度・回数の上限/下限は定義されているか
- エラー時・例外時のフローは記述されているか

#### D. 関心の分離

- 1つのAPIエンドポイントに複数の責務が詰め込まれていないか
- 1つの画面操作が複数の無関係な副作用を引き起こしていないか

#### 質問の方針

- 表面的・自明な質問は避ける。仕様を読めばわかることを聞かない
- 選択肢を提示して、ユーザー自身も気づいていない判断基準や制約を引き出す
- UI/UX、エッジケース、トレードオフに関する質問は特に重要であり、積極的に行う
- 回答を受けたら、その回答から派生する新たな疑問がないか考え、あれば続けて質問する

#### 質問の例

良い質問（選択肢で判断を引き出す）
- 「このステータス遷移で、完了→到着に戻したとき、API再送信について (A) 再送信する (B) 送信済みなら何もしない (C) エラーにする のいずれが適切ですか？」
- 「オンライン診療の主訴表示について (A) 対面と統合して受診方法を後から選択 (B) 別セクションに分離 (C) フィルタで切り替え のどれがユーザー体験として自然ですか？」

悪い質問（仕様を読めばわかる、または漠然としすぎ）
- 「メールを送る機能は必要ですか？」
- 「何か不明点はありますか？」

### 3. 設計方針の合意

対話を踏まえ、実装の設計方針を簡潔にまとめる。

```
## 設計方針

### アプローチ
[選択したアプローチとその理由]

### 前提条件
[合意した前提。仮の前提には「仮」と明記]

### スコープ
[この実装で対応するもの / しないもの]
```

この合意を得てから実装に着手する。

### 4. 対話で得た知見の保存

対話を通じて将来の会話でも価値のある知見が見えた場合、メモリに保存する。
保存対象の例
- 設計思想・哲学（ドメイン分離、冪等性など）
- UX方針（選択肢の出し方、画面遷移の好みなど）
- ビジネスルール・ドメイン知識（判定ロジック、業務フローなど）
- 技術的制約（外部APIの仕様、システム間の制約など）
- ステークホルダー情報（誰が何を決めるか）

保存ルール
- ファイル名: `design_[テーマ].md`（設計系）、`project_[テーマ].md`（プロジェクト系）、`user_[テーマ].md`（ユーザー系）
- 保存先: `~/.claude/projects/*/memory/`
- frontmatter に `created` と `decay` を必ず含める
  - 設計哲学・行動原則: `decay: permanent`
  - アーキテクチャ判断・技術選定: `decay: 6months`
  - プロジェクト固有の判断・ビジネスルール: `decay: 3months`
  - 一時的な制約・暫定対応: `decay: 1month`
- 既存のメモリと重複しないか確認してから保存する

## 注意事項

- このスキルは「実装を遅らせる」ためのものではない。「正しい方向に走り出す」ためのもの
- 仕様が完璧に見えても、状態遷移と冪等性は必ず質問する
