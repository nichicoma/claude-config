# コードレビュー

## 前提

コードレビューは「バグを見つける作業」ではない。
「この変更は自分たちのプロダクトに組み込むべきか」を判断する行為である。

機械的な検査（lint、型チェック、テスト）は自動化の責務。
人間のレビュアーが担うべきは **判断（Judgment）** — 設計の妥当性、意図の明確さ、プロダクトとの一貫性。

## 実行手順

### 1. 差分の取得

- `$ARGUMENTS` にブランチ名が指定されている場合: `git diff main...$ARGUMENTS` で差分を取得
- `$ARGUMENTS` にファイルパスが指定されている場合: そのファイルの差分を取得
- `$ARGUMENTS` が空の場合: `git diff HEAD` で全変更（staged + unstaged）を取得。差分が空なら `git log --oneline -1` で直近のコミットとの差分を確認

### 2. 変更の全体像を把握

差分を読む前に、以下を確認する。

- 変更されたファイルの一覧と変更行数
- コミットメッセージ（あれば）
- 関連する issue や PR の説明文

変更の「意図」を先に理解してから、コードを読む。意図が不明な場合は AskUserQuestion で確認する。

### 3. レビュー観点チェックリスト

以下の6つの次元で評価する。すべてに言及する必要はなく、問題がある次元のみ報告する。

#### A. 設計の意図（Intent Assessment）

- 変更の目的が差分から読み取れるか
- 「なぜこうしたか」が不明な箇所はないか
- コミットメッセージや PR 説明と実装に矛盾がないか

#### B. 品質基準（Taste and Standards）

- CLAUDE.md の設計原則（SRP, DRY, KISS, SoC）に違反していないか
- TypeScript の strict ルール（`any` 禁止、`as` 回避、型推論優先）を満たしているか
- コンポーネント設計パターン（Composition / Compound）の適用は適切か
- 命名は目的を明確に表しているか

#### C. アーキテクチャの方向性（Architecture Velocity）

- 既存のアーキテクチャと整合しているか
- 依存方向（下位→上位の一方向）を破っていないか
- 新たな循環依存を生んでいないか

#### D. リスク評価（Risk-Based Routing）

- 認証・認可・データ削除など高リスク領域の変更に十分な注意が払われているか
- 外部 API や依存パッケージの変更がある場合、影響は評価されているか
- ロールバック可能か

#### E. プロダクトの一貫性（Product Coherence）

- この変更はプロダクト全体の方向性と一致しているか
- ユーザー体験に予期しない変化をもたらさないか
- アクセシビリティ要件を満たしているか

#### F. メンタルモデルの存在確認（Theory Building Check）

- コードの背後に「なぜこうなっているか」の理解（メンタルモデル）が存在するか
- 以下の兆候がある場合、メンタルモデル不在の「検証負債」を疑う
  - 動くが理由が説明できないコード
  - 過剰に防御的なエラーハンドリング（何が起きるかわからないから全部 catch）
  - コピー元が不明な定型コード
  - テストはあるが、テスト対象の振る舞いの「なぜ」が説明できない
  - 不自然に冗長な実装（シンプルな代替案が存在するのに選ばれていない）

### 4. レビュー結果の報告

以下の形式で報告する。

```
## レビュー結果

### 総評
[1-3文で変更全体の評価。問題がなければ承認の意思を明示]

### 指摘事項
[問題がある場合のみ。以下のプレフィックスを使用]

- **MUST**: 修正が必要。マージ前に対応すべき問題
- **SHOULD**: 強く推奨する改善。合理的な理由があれば据え置き可
- **NIT**: 細かい指摘。対応は任意
- **IMO**: 個人的な意見。議論の余地あり
- **FYI**: 参考情報。対応不要
- **QUESTION**: 意図の確認。理解のための質問

### 良い点
[具体的に良いと思った実装判断があれば記載。お世辞は不要]
```

### 5. 対話による深掘り

- 設計意図やトレードオフが不明な箇所は、AskUserQuestion で質問する
- 質問は「なぜこうしたのか」を理解するためのものであり、詰問ではない
- 回答を踏まえてレビュー結果を更新する

## 注意事項

- 「バグがないこと」の保証はこのレビューの目的ではない。それはテストの責務
- 重箱の隅をつつくレビューは避ける。重要な判断に集中する
- 指摘がゼロでも問題ない。「問題なし」は立派なレビュー結果
- レビューコメントは人格ではなくコードに向ける
