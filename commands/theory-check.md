# Theory Building チェックポイント

## 前提

Peter Naurの「Programming as Theory Building」によれば、プログラムの本体はソースコードではなく、プログラマの頭の中にあるメンタルモデルである。
メンタルモデルが失われたコードは「死んだプログラム」— 動きはするが、なぜそうなっているかを誰も説明できない状態になる。

AI生成コードは、このメンタルモデルが最初から存在しないリスクを持つ。
このチェックポイントは、実装者がコードを「自分のもの」にするための確認プロセスである。


## 実行手順

### 1. 変更の全体像を把握

```bash
git diff --stat
git diff
```

変更されたファイル、追加行数、削除行数を確認する。

### 2. 設計判断サマリの作成

変更内容を分析し、以下を整理する。

```
## 設計判断サマリ

### 何を作ったか
[変更の概要を1-3文で]

### 主要な設計判断
[判断ポイントごとに、選択肢と選んだ理由を記述]

1. **[判断ポイント]**
   - 選択: [選んだアプローチ]
   - 理由: [なぜこれを選んだか]
   - 代替案: [検討した他の選択肢。なぜ選ばなかったか]

### 影響範囲
[この変更が波及する箇所]

### 制約・前提
[この実装が依存している前提条件]
```

### 3. 確認質問の提示

AskUserQuestion を使い、実装者のメンタルモデルを検証する質問を1つずつ対話的に確認する。
サマリ提示後、必ず最初の質問を投げかける。質問ゼロで完了としない。
回答から新たな論点が見えたら、継続的にインタビューする。

#### 質問の設計方針

- 表面的・自明な質問は避ける。コードを読めばわかることを聞かない
- コードの「何」ではなく「なぜ」を問う
- 選択肢を提示して、実装者が自分の理解を言語化する手助けをする
- 将来の変更シナリオに対する予測を問う

#### 質問カテゴリ

**設計意図の確認**
- 「この部分で [A] ではなく [B] を選んだ理由は (1) パフォーマンス (2) 可読性 (3) 既存パターンとの一貫性 (4) その他 のどれに近いですか？」
- 「[エンティティX] にこの責務を持たせた根拠は何ですか？」

**変更耐性の確認**
- 「もし [要件の変化] が起きた場合、どこを変更する必要がありますか？」
- 「この機能を [拡張シナリオ] に対応させるとき、現在の設計で (A) そのまま対応可能 (B) 小規模な修正が必要 (C) 設計の見直しが必要 のどれですか？」

**境界の確認**
- 「この変更のスコープ外としたものは何ですか？その判断の理由は？」
- 「[関連する別機能] との境界はどこに引きましたか？」

### 4. 結果の判定

対話の結果に応じて判定する。

- **メンタルモデルが形成されている**: 設計判断の理由を自分の言葉で説明でき、変更シナリオに対する予測ができる状態。コミット/PR作成に進んでよい
- **部分的に不明確**: 特定の箇所について理解が曖昧。該当箇所のコードを一緒に読み直し、理解を深めてから進む
- **メンタルモデルが不在**: 設計判断の理由が説明できない、または「AIが書いたのでわからない」状態。コードを読み直し、必要に応じて書き直す

### 5. 対話で得た知見の保存

対話を通じて将来の会話でも価値のある知見が見えた場合、メモリに保存する。
保存ルールは @~/.claude/commands/shared/memory-save-rules.md を参照

## 注意事項

- これは試験ではない。「わからない」は恥ずかしいことではなく、重要な情報
- 質問数は3-5個に絞る。網羅性より、核心的な判断ポイントに集中する
- 回答が不十分でも責めない。一緒にコードを読み直して理解を深める姿勢で
- 実装者が「もう十分理解している」と判断した場合、途中で切り上げてよい
